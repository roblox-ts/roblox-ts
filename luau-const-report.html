Implementing the Luau const keyword in roblox-ts

An investigation into feasibility, risks, and the variables we declare-then-reassign.
Repository: roblox-ts/roblox-ts · Generated 2026-06-30 · Investigation only — no code changed.
TL;DR. Adding const is feasible and mostly mechanical, but it cannot be a blanket rename of local. Luau const requires an initializer and forbids any later reassignment (including compound assignment and reassignment of captured upvalues from nested functions). roblox-ts has many codegen patterns that deliberately declare a variable and reassign it later — your instinct about optional chaining and destructuring defaults is correct, and there are several more (ternaries, &&/||/??, hoisting, class/enum scaffolding, for loops). The safe path is an analysis pass that only promotes declarations that are provably never reassigned, rather than auditing 40+ emit sites by hand.

1. What Luau const actually guarantees

From the Luau const-keyword RFC and language docs, const is a contextual keyword usable anywhere local is. The semantics that constrain us:

PropertyRuleConsequence for codegen
InitializerMandatory. A const binding must be initialized at the point of declaration.Any local x with no right-hand side can never become const.
ReassignmentForbidden in all forms — simple =, compound += etc., and reassignment of a captured upvalue from inside a nested function.If we ever emit an Assignment targeting the binding, it must stay local.
Multiple assignmentSupported: const a, b = f().Tuple declarations are eligible — but only if none of the names is later reassigned.
Binding vs valueOnly the binding is immutable; the referenced table can still be mutated.Tables we build up via t.x = ... after declaration are fine to mark const (the binding isn't reassigned).
Shadowing / redeclarationAllowed.No conflict with roblox-ts's existing shadowing behavior.
MotivationClarity, intent, stable exports, better tooling/type-checker reasoning. The RFC explicitly says it is not a performance feature.This is a readability/correctness-signal change, not an optimization. Worth weighing against added compiler complexity.

2. How roblox-ts emits locals today

Every local in the output comes from a single AST node, luau.SyntaxKind.VariableDeclaration, defined in the @roblox-ts/luau-ast package (currently pinned to =2.0.0 in package.json): 

// node_modules/@roblox-ts/luau-ast/.../nodes.d.ts
interface VariableDeclaration extends BaseStatement<SyntaxKind.VariableDeclaration> {
    left: AnyIdentifier | List<AnyIdentifier>;
    right: Expression | List<Expression> | undefined;   // undefined ⇒ `local x` with no value
}

It is rendered by renderVariableDeclaration — note that the keyword local is hard-coded here:

// node_modules/@roblox-ts/luau-ast/.../renderVariableDeclaration.js
if (node.right) {
    return state.line(`local ${leftStr} = ${rightStr}`, node);
} else {
    return state.line(`local ${leftStr}`, node);   // no initializer
}
Key structural fact: the AST node and its renderer live in the separate, published @roblox-ts/luau-ast dependency — not in this repo. Emitting const requires a change there (a flag on the node + a renderer branch), a new release, and a version bump here. Everything in roblox-ts itself is just deciding when to set that flag.

Aside: TypeScript's three declaration forms are already collapsed. var is rejected (errors.noVar in src/Shared/diagnostics.ts), and both let and const currently compile to the same Luau local. So the TS keyword the user wrote is not currently a reliable signal on its own — see caveats.

3. Declare-then-reassign sites — these must stay local

These are the patterns where roblox-ts intentionally creates a binding and writes to it afterward. Every one of them would be a compile-to-broken-Luau bug if naively emitted as const. This is the heart of your concern, and it is broader than just destructuring/optional chaining.

PatternWhereConst?Why it reassigns
Optional chaining a?.b?.c() transformOptionalChain.ts
(createOrSetTempId ~L199/L207)		 never Declares a temp, then reassigns it inside each if x ~= nil then … nil-guard as the chain advances.
Destructuring with defaults { a = 5 } = obj transformInitializer.ts
(via transformObject/ArrayBindingPattern)		 never Emits local a = obj.a then if a == nil then a = 5 end. The default is a reassignment.
Optimized array destructure [a, b] = f() transformVariableStatement.ts L94
transformBinaryExpression.ts L97		 never Declares local a, b with no value, then a, b = … — fails the initializer rule outright.
Ternary cond ? x : y (with prereqs) transformConditionalExpression.ts L37 never local result (no value) then result = … in both if/else branches.
Logical chains && / || / ?? (non-inlined) transformLogical.ts L69 never First link is local condition = a; later links reassign condition = b inside nested guards.
Hoisted variables (used before declaration) transformVariableStatement.ts L47–51
createHoistDeclaration.ts L11		 never local x is emitted at the top of the scope and the real value is assigned later via x = ….
Class declarations (hoisted) transformClassLikeDeclaration.ts L240 never local Class with no value, populated afterward.
Enum scaffolding transformEnumDeclaration.ts L125 never local Enum declared empty, then filled by member assignments.
for loop control / increment flags transformForStatement.ts L217
transformForOfStatement.ts		 never e.g. local _shouldIncrement = false → reassigned to true; numeric loop vars are mutated each iteration.
Compound assignment temps, spread-destructure generator, iterable counters transformUnaryExpression.ts, spreadDestructureGenerator.ts, getAddIterableToArrayBuilder.ts never Counters/accumulators that are incremented or reassigned by design.
The trap: "has an initializer ⇒ make it const" is not safe. Several of the cases above (logical chains, _shouldIncrement = false) declare with a value and still reassign later. Initializer presence is necessary but not sufficient.

4. Const-eligible sites — the safe win

The large majority of generated locals are write-once. These are the ones worth promoting:

PatternWhereConst?
Plain const x = … / let x = … never reassigned, not hoistedtransformVariableStatement.ts L53 (transformVariable)eligible
Simple destructure field without a default — local a = obj.atransformObjectBindingPattern.ts / transformArrayBindingPattern.tseligible
RuntimeLib import — local TS = require(…)TransformState.ts (createRuntimeLibImport) L225+eligible
Import / export declarationstransformImportDeclaration.ts L59, transformExportDeclaration.ts L57eligible
Cached complex sub-expressions — pushToVarIfComplex with a value, never written againTransformState.ts (pushToVar / pushToVarIfComplex)usually
next() key/value temps in map destructuringgetAccessorForBindingType.ts (mapAccessor)eligible

"usually" = eligible whenever that particular temp is not subsequently used as an Assignment target. This is exactly the distinction the promotion pass below makes automatically.

5. Recommended implementation strategy

Two layers of change, and a clear choice on the decision logic.

Layer A — @roblox-ts/luau-ast (the dependency)

  1. Add an optional flag to the node, defaulting to local for backward compatibility: 
    interface VariableDeclaration … {
    left: AnyIdentifier | List<AnyIdentifier>;
    right: Expression | List<Expression> | undefined;
    isConst?: boolean;   // NEW
}
  2. In renderVariableDeclaration, choose the keyword — and assert the initializer invariant so a bad promotion fails loudly in tests rather than producing invalid Luau: 
    const keyword = node.isConst ? "const" : "local";
if (node.isConst) assert(node.right !== undefined, "const requires an initializer");
return node.right
    ? state.line(`${keyword} ${leftStr} = ${rightStr}`, node)
    : state.line(`${keyword} ${leftStr}`, node);
  3. Release a new luau-ast version and bump the =2.0.0 pin in this repo's package.json.

Layer B — roblox-ts (deciding when it's const)

Two viable approaches:

ApproachHowTrade-off
Per-site opt-in Hand-set isConst: true only at the known-safe emit sites from §4. Simple, incremental, low risk — but covers only the sites you remember to touch, and every future codegen change risks introducing an unsafe one.
Promotion pass (recommended) Leave all emit sites alone. After transform, run one analysis pass that promotes every declaration proven write-once. Correct by construction for all of §3 automatically (those bindings are assigned later, so they stay local); slightly more upfront work, one place to reason about.
Recommendation: the promotion pass. It makes the §3 footguns impossible to hit — a binding that is ever an Assignment target is simply never promoted — and it doesn't require auditing or babysitting 40+ creation sites. The per-site approach can ship first as a smaller stepping stone if desired.

6. The promotion pass in detail

Run once over each source file's finished Luau AST, just before rendering (e.g. invoked from transformSourceFile):

  1. Collect assignment targets. Walk the whole tree and record every identifier that appears on the left of an Assignment (any operator, including compound) — this also covers writes that happen inside nested function bodies, which is required because Luau forbids reassigning a captured const upvalue.
  2. Promote. Walk again; for each VariableDeclaration, set isConst = true iff:
    • right is defined (initializer present), and
    • none of the identifiers in left (whether a single id or a tuple) is in the assignment-target set.

Worked example — what the pass produces:

-- write-once: promoted
const player = Players.LocalPlayer
const TS = require(game:GetService("ReplicatedStorage").rbxts_include.RuntimeLib)

-- optional chain temp: reassigned in the guard, stays local
local _result = a
if _result ~= nil then _result = _result.b end

-- destructure default: reassigned, stays local
local count = opts.count
if count == nil then count = 10 end

-- ternary: declared empty then assigned, stays local
local _result
if cond then _result = x else _result = y end

7. Caveats & edge cases

  • Identity-based matching, not name-based. roblox-ts already uses unique TemporaryIdentifier objects and renames collisions only at render time. Match assignment targets by identifier identity/symbol where possible. If you fall back to name matching, ensure errors bias toward keeping local (a false "is assigned" is safe; a false "not assigned" is a bug). Over-matching is the safe direction.
  • Nested-function captures. Because a captured const can't be reassigned from an inner closure, the assignment-target scan must be whole-tree, not per-scope. The pass above already does this.
  • export let is not a local. Mutable exports compile to module-table field writes (transformVariable L28–42), not local, so they're outside this feature entirely — no action needed.
  • TS let can still be write-once. A let the user never reassigns will be correctly promoted by the pass. Conversely a TS const is never reassigned at the TS level, but roblox-ts may still reassign a related compiler temp — so trust the Luau-AST analysis, not the TS keyword.
  • Opt-in / config. Since the RFC frames const as a clarity feature (not perf) and it changes output for nearly every file, consider gating it behind a compiler option (e.g. luau.const in tsconfig) at least initially, and confirm the minimum Luau/runtime version your users target supports the keyword.
  • Test surface. The compile-snapshot tests under tests/ will churn heavily. That's expected and is actually the best validation that promotions landed exactly where intended.

8. Suggested rollout plan

  1. Land the isConst flag + renderer branch + assert in @roblox-ts/luau-ast; release and bump the pin.
  2. Implement the promotion pass behind an off-by-default flag.
  3. Regenerate compile snapshots; spot-check the §3 patterns (optional chain, destructure defaults, ternary, logical, hoists, enum/class, loops) to confirm they remain local.
  4. Run the full Luau test suite (test-run / run-in-roblox) to confirm runtime parity.
  5. Decide whether to flip the flag on by default or keep it opt-in.
